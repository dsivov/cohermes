the starting point

The loop you already use

Your development methodology is design-first and human-in-the-loop:

  1. Discuss — we open an illustrated HTML discussion doc and think out loud in it.
  2. Iterate → agree — a few rounds until the shape is right.
  3. Architecture & work plan — we write the arch docs and a concrete plan.
  4. Code — only then do we build.

The value is in the first three steps — the shared understanding — and the coding is the (comparatively mechanical) payoff. The question is how to keep that when the "developer" is now a coordinator delegating to opus over a one-shot call.

keythe principle

Delegate the coding — not the design

A one-shot claude -p is the wrong tool for a discussion: that phase is creative, iterative, and human-in-the-loop — it needs you and opus going back and forth, exactly like your loop today. So the split is by phase, not by tool:

PhaseModeWhy
Discuss · iterate · agreeinteractivecreative, back-and-forth, yours to steer
Architecture · work planinteractivestill a conversation — you shape it
Code · executedelegatedexecuting an already-agreed plan
The one idea
cohermes keeps the thinking interactive and makes the shared brain (Context Graph) carry each phase's artifacts — the discussion doc, the decisions, the work plan — so every later step, and every teammate, builds on the same agreed context instead of re-deriving it.
flowthe mapping

Phase by phase in cohermes

Each phase writes to CG; each next phase (and each teammate) reads from it. The brain is the loop's memory.

1Discuss. Work interactively with opus (the Claude Code flavor, or `cohermes agent`) to produce the illustrated HTML discussion doc — the same as your loop today. When it settles, ingest it into CG (so it's retrievable) and record the key decisions as they land.

2Architecture & work plan. Still interactive. Capture the architecture, and record the plan as governed artifacts — ADRs (the why) and a set of Tasks (the what) — via the workspace's governed actions. Now the plan is shared, queryable state, not a doc someone has to remember.

3Code. Delegate execution to opus. For a quick change use ask_claude; for a real implementation you want to watch, use delegate_claude(write=true) (§5). Opus is grounded in the CG artifacts from steps 1–2 — it reuses the agreed decisions instead of re-deriving — and its outcome is recorded back to CG.

why CGthe substrate

Context Graph is the connective tissue

What makes this a loop rather than four disconnected steps is that every artifact lives in one shared, queryable place:

  • the discussion doc → ingested prose (retrievable context);
  • the decisions → ADRs / decision traces (precedent-searchable);
  • the work plan → Tasks with state.

So when the coding phase runs, opus is handed the relevant precedent automatically (the delegation tools inject a CG brief and record the outcome back — see How Context Works). A teammate picking up the next task reads the same brain. Nothing agreed is lost, and nothing is re-litigated.

§5observability

Watching the execution — you're never blind

The gap in a plain -p delegation is that it's opaque. cohermes closes it with a tmux delegation: opus runs in a session you can attach to and watch in real time.

You want to see…How
Opus working live — its reasoning, tool calls, todo listdelegate_claude(...) → tmux attach -t <session>
Which files it editedthe tmux stream (Edit/Write calls) · or watch -n2 git status
Which task is in progressopus's TodoWrite in the tmux stream
The result when it's donecheck_claude('<session>') (also records it to CG)
The decision/task historythe CG WebUI, or search_precedents / orient
Non-blocking too
Because delegate_claude returns immediately and opus runs in the background, a long implementation never blocks the session or trips a timeout — you delegate, watch, and check back.
who does what

The two roles behind it

Under the hood, execution uses the coordinator + worker split (see Setup):

RoleRuns onDoes
Coordinatora cheap model (local / Bedrock)routes, talks to CG, delegates — and relays the tmux attach line to you
Workeropus — the real claude binaryall the actual design/plan/code, first-party (plan limits)

For the design phases you can skip the coordinator entirely and talk to opus directly (the Claude Code flavor) — the coordinator earns its keep on the delegated execution.

exampleend to end

Worked example — a feature, the cohermes way

  1. Discuss. Interactively, you and opus write docs/provenance-origin-tracing-discussion.html — trace each claim to its earliest source, grade the origin by trust. Iterate until agreed; ingest the doc into CG.
  2. Agree & record. Record the ADRs (e.g. build it as a read-time extension of corroboration.py) and the work plan as Tasks in CG.
  3. Code. Ask the coordinator to implement it. It calls delegate_claude(task, write=true) and hands you tmux attach -t coh-…. You attach and watch opus — grounded in the recorded decisions — create graph/provenance.py, wire it in, add tests. When it finishes, check_claude relays the summary and records the outcome to CG.
  4. Next. The recorded decision + outcome now feed the next task's orient — for you or a teammate.
cheat sheet

Quick reference

# DESIGN phases — interactive opus (like your loop today), then record to CG
cd project && claude            # discuss the HTML doc, architect, plan — iterate
# … then have it ingest the doc + record ADRs/Tasks to Context Graph

# EXECUTION phase — delegate + watch
# (through the coordinator; it will call these for you)
delegate_claude(task, write=true)   # launches opus in tmux, returns a session name
tmux attach -t coh-<id>              # watch opus work live (detach: Ctrl-b then d)
check_claude("coh-<id>")            # fetch the result when done (records it to CG)
ask_claude(task)                    # quick, blocking — for short read-only work
