why this matters

The question

When a developer works with a Claude Code agent, the whole session is part of the success: the agent knows what was done, where things are, how to start the dev server, which port a service runs on. A fair worry: in a cohermes/Hermes setup, where does the working agent get all of that? Only from the Context Graph? Is that enough for complex development?

Short answer
The Context Graph is one layer — the durable, shared one. A working agent assembles four layers, and cohermes makes exactly one of them first-class. CG is not, and was never meant to be, the whole context.

the honest answer

Is the Context Graph enough on its own?

No — and that's by design. CG's job is to make sure the agent doesn't re-derive team decisions and starts aligned. Complex development still needs the operational memory (CLAUDE.md), the live code (read on demand), and the session's working state. cohermes makes the shared, durable layer excellent; the rest stays Claude Code's native context.

What converts a session into shared knowledge
The loop — record-the-why, record_learning, ingest_context. That's how the durable parts of this session (a decision, a gotcha, "the server runs on :9621") become layer A/B for the next agent. What you don't promote is re-derived or lost — inherent to any hand-off. cohermes maximizes what survives; it can't perfectly clone a session.

So the completeness of the shared context depends on team discipline in promoting knowledge — which the tools + orient make cheap, but the team still has to do.

the runtime

Who decides to call tools, subagents, and MCP?

The model (Claude), inside Claude Code's agentic loop — not Hermes. Hermes is the outer orchestrator; it launches and manages Claude Code but does not drive its inner tool loop.

who reasons

The coordinator/worker split — where the thinking actually happens

In the Hermes-runtime flavor there are two models, and they are deliberately unequal. A cheap coordinator runs the outer loop; the real reasoning is delegated to opus. The reason is not elegance — it's a 2026-07 billing correction that forced the split.

The 2026-07 metering correction
Anthropic now routes third-party apps that authenticate with your subscription OAuth token through "extra usage" (a paid per-token overage), not your plan limits. Hermes's own agent loop calls the Messages API with a claude-code (external, cli) user-agent + your token — a third-party call — so Hermes-as-a-model is metered. Only the real first-party claude binary draws from plan limits. So all substantive reasoning must run through the actual claude CLI, never through Hermes-as-a-model.

That correction is what defines the roles. The coordinator handles only the cheap, high-frequency work; the worker does everything that counts:

CoordinatorWorker
Modela cheap model (e.g. Bedrock Gemma 4 31B, or a local llama.cpp/Ollama)opus — the real claude binary
Doesrouting · deciding what to delegate · Context Graph interaction (orient / precedent / record via the context-graph MCP tools)all substantive reasoning — plans, architecture, design, decisions, code
Billingthe one accepted metered piece (cheap model, light role)first-party · plan limits — never metered

The coordinator never produces the substantive answer itself; it delegates through one tool, ask_claude (in cohermes's MCP server), which runs claude -p as a subprocess with list-args (no shell-quoting bugs) and returns opus's output. Delegation is enforced in code, not by prompt — the coordinator's file/terminal/code tools are stripped, so a weak model cannot quietly do the work (and meter it) instead of handing it to the binary.

How this maps onto the four layers
The coordinator is the one reading and writing layer A (Context Graph) — orienting, querying precedent, recording decisions — and routing. But the reasoning that consumes A + B + C in a finite window is opus's, inside its own first-party claude loop. The split is a billing boundary drawn through the architecture: cheap coordination on the metered side, all real thinking on the plan-limits side.

the constraint

The window is finite — so curation is architectural

Everything converges on one finite context window, which the injected knowledge shares with the code the agent must read. You cannot pour the whole team brain into it. That's precisely why the brain is curated (store the durable why, not a repo mirror) and orient returns a focused brief, not everything.

Your "curated learnings, not index-overload" was load-bearing
It isn't just tidiness — it's what keeps the working agent's window high-signal. The design that follows from the constraint: a curated brain + a focused orient brief + on-demand retrieval (the agent pulls more via tools only when it needs it), rather than front-loading everything.
the subtlety

Does -p start empty each task? No.

claude -p starts a fresh conversation — but not an empty context. And it is not Hermes that accumulates and feeds the layers.

On every -p call, the durable layers are re-established from their own persistent homes — no session needed to carry them:

LayerOn each -p call…from
B operationalauto-loaded by Claude Code at startupCLAUDE.md (git)
A shared knowledgeinjected via --append-system-prompt (cohermes-agent)CG (via orient)
C codebaseread on demand via toolsgit
D session stateresets — the one thing that doesn't carry over(the previous session — gone unless --resume)
Why statelessness is fine
The important context doesn't live in the session — it lives in durable stores (CG, CLAUDE.md, git) that reconstitute A+B+C on every call. So a fresh -p agent is never empty; it's freshly rebuilt from durable sources, plus a clean D for the task. Need D to survive? --resume. cohermes's way: promote D's durable bits to CG so they come back as A next time.
a common misconception

Where does conversation history actually live?

It is stored on disk — just not by the model. Three distinct levels, and separating them dissolves the confusion:

The one-liner
The model is stateless; Claude Code is what stores conversation history on disk and re-sends it each turn; Hermes keeps its own. cohermes puts the knowledge that must outlive and cross sessions (and machines) in CG — not in any transcript, which is local, per-agent, and transient.
a fair question

Isn't this just Hermes's memory, in graph mode?

Hermes already tracks sessions, recalls across sessions (FTS5 search), and bookkeeps async work. So does the Context Graph just re-implement that? No — it's a different kind of memory for a different purpose; they barely overlap.

The clincher
The whole reason cohermes exists is that Hermes's memory is per-user and local — the Hermes source itself states "memory is NOT replicated across instances or machines." It structurally cannot be a team brain. CG fills exactly that gap; it doesn't duplicate it. And the async/cron/kanban bookkeeping is Hermes's own runtime job queue — orthogonal to a shared knowledge graph, not something CG re-does.

The analogy: Hermes's state.db is your private notes and chat log with an assistant; CG is the team's engineering decision records + architecture graph. One is "what I discussed"; the other is "what the team knows." cohermes runs them side by side.

The mental model

Keep these three sentences and the rest follows:

  • Context is four layers — durable shared (CG), operational (CLAUDE.md), the code (git), the session (transient). cohermes owns the first; the others are native.
  • The model drives the tool/MCP/subagent calls inside Claude Code; Hermes orchestrates from outside and never touches that inner loop.
  • Everything converges on a finite window, so the brain stays curated and orient stays focused — the agent retrieves more only on demand.
Net
cohermes doesn't try to be the whole context. It makes the one layer that was missing — durable, shared, cross-agent knowledge — first-class and cheap to read/write, and lets Claude Code's native layers do the rest.
