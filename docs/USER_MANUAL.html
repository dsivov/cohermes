What cohermes is now

cohermes is Claude Code + a shared Context Graph, with an optional Hermes coordinator in front. Whatever flavor you run, the same rule holds: first-party opus does all the real work — plans, architecture, design, decisions, and code. The Context Graph is the team brain: durable, cross-session context that every agent reads from and records back to.

The Hermes coordinator, when you use it, is deliberately small. It routes, decides what to delegate, and talks to the Context Graph — it never writes the substantive answer itself. It hands the actual reasoning to the real claude binary running opus.

The one rule
All real reasoning runs through the actual first-party claude CLI (opus), never through Hermes acting as a model. Only the genuine claude binary draws from your plan limits — everything else is either a cheap coordinator or an overage bill.
1pick a flavor

The two ways to run

Both flavors do the actual reasoning first-party on opus. They differ in what sits in front of opus.

(a) Claude Code flavor — opus is the agent

You run the real claude (via cohermes agent / cohermes-agent) and opus is the agent directly. There is no coordinator layer. This is the best flavor for interactive design and planning — sitting with the agent, thinking about your project turn by turn. First-party, plan limits.

(b) Hermes-coordinator flavor — cheap coordinator + opus workers

A cheap coordinator model fronts Hermes's orchestration (tools, memory, gateway, scheduling) and delegates every substantive task to an opus worker through ask_claude. Best for automation and multi-agent orchestration where you want something cheap deciding what to run and opus doing the work.

Rule of thumb
Reach for (a) when a human is in the loop designing. Reach for (b) when you want cheap automation to drive opus without a person at the keyboard. Either way the substantive answer comes from first-party opus.
2the rhythm

The loop: orient → work → record

Every session follows the same three beats, whichever flavor you run.

  • Orient — pull the team's context from the Context Graph (past decisions, precedent, learnings) so the agent starts aligned instead of re-deriving. In flavor (b) the coordinator does this via the context-graph MCP tools.
  • Work — opus does the substantive task: the plan, the design, the code. This is always first-party opus, never the coordinator.
  • Record — write the decisions and learnings back to the Context Graph so the next session (yours or a teammate's) inherits them.
3flavor (b) only

Configure your coordinator (per-developer)

The coordinator is a per-developer choice — your own hardware and keys. It lives in the project-local, gitignored <project>/.hermes/config.yaml, never in the committed config. (Skip this section entirely if you run flavor (a).)

Recommended: Bedrock Gemma 4 31B

Hosted, fast (~6–10s/turn), and capable enough for a light routing role. It's an OpenAI-compatible endpoint:

model:
  provider: custom
  default: google.gemma-4-31b
  base_url: https://bedrock-mantle.us-east-1.api.aws/openai/v1   # regions: us-east-1/2, us-west-2, eu-central-1
  api_key: <Bedrock API key / bearer token>
  context_length: 262144            # 256K
  extra_body: { parallel_tool_calls: false }   # REQUIRED — Bedrock Gemma 4 has no parallel tool calls; omit and it hangs
Required: parallel_tool_calls: false
Bedrock Gemma 4 has no parallel tool calls. Leave parallel_tool_calls: false in extra_body — omit it and the coordinator hangs.

Auth is a Bedrock API key (a bearer token) used as the OpenAI api_key. Note: a box's ambient IAM credentials may lack Bedrock permissions — use the bearer token, which is account-scoped.

Alternative: a local model

A local model via llama.cpp or Ollama works too, but it must serve a ≥64K context window — that's Hermes's floor. For example:

llama-server --ctx-size 65536

Point base_url at it. Viable, but slower and weaker than Bedrock Gemma 4 31B (the local 12B was too weak and too slow).

4run

Running it

Launch the Hermes-coordinator flavor from the project, pointing HERMES_HOME at the project's own .hermes/ home:

HERMES_HOME=<project>/.hermes  /storage/Work/cohermes/.venv/bin/hermes chat

That uses the durable venv, the per-developer .hermes/ home, and the Context Graph MCP pinned to the project's workspace. Each project keeps its own .hermes/ — that's where your gitignored config.yaml (coordinator model) and role policy live.

Per-project home
<project>/.hermes/ holds the coordinator config, the SOUL.md role policy ("you are a coordinator; use ask_claude for all real work"), and session state. It is per-developer and gitignored — your teammates each keep their own.

For flavor (a), you don't run hermes chat at all — you run the real claude (via cohermes agent) and opus is the agent directly. See the Setup Guide for wiring.

5the money

What runs where / who pays

PieceModelWho pays
Coordinator (flavor b)cheap model (e.g. Bedrock Gemma 4 31B)small metered cost — the one accepted paid piece
Worker (all real work)opus, first-party claudeplan limits — no per-token bill

The worker is where all the substantive reasoning happens, and it stays first-party opus — drawn from your plan limits, not billed per token. The coordinator is a cheap model doing a light role, so its metered cost is small. That split is the whole point: keep the expensive reasoning on plan limits, pay only pennies for routing.

if something's off

Troubleshooting

SymptomFix
Coordinator hangsBedrock Gemma 4 has no parallel tool calls — set extra_body: { parallel_tool_calls: false } in your .hermes/config.yaml.
Local model misbehaves / truncatesIt needs a served context window of ≥64K (Hermes's floor). Serve it with e.g. llama-server --ctx-size 65536.
"Extra usage" HTTP 400You routed real work through Hermes's own agent loop instead of the claude binary. Hermes-as-a-model is a third-party call that bills extra-usage; make sure substantive work goes through ask_claude → the real first-party claude, which draws from plan limits.
Why the 400 happens
Since Anthropic's 2026-07 policy change, third-party apps that authenticate with your subscription OAuth token draw from paid "extra usage", not plan limits. Hermes's own loop is such a third-party call, so it bills overage (400 when the balance is 0). Only the genuine claude binary is first-party — keep all real reasoning there.
