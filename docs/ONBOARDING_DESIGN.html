Why we're here

The problem: onboarding is artisanal

Bringing the investigator project into cohermes worked — but by hand. I wrote two launcher scripts (cg/hermes, cg/cohermes-agent), a .hermes.md working agreement, and a project-local Hermes home config, and threaded the workspace name through all of them. Every one of those steps encodes a lesson that a new project has no way to know.

That doesn't scale, and it buries hard-won knowledge in per-repo bash. A fresh project starts blank: it doesn't know which CG workspace holds its memory, how to reach Claude on the subscription (not metered), or that this team records decisions through governed actions. Onboarding has to supply that knowledge, consistently, without the operator re-deriving it each time.

The goal
A project should become self-describing to cohermes — declare a little, inherit the whole methodology. The knowledge lives in the tool, not copied into every repo.
The hard part

What onboarding must encode — the six traps

Universal onboarding is only worth building if it bakes in the traps I hit by hand. Each one silently breaks a project that a naïve setup would leave for the next person to rediscover.

TrapWhat goes wrongEncoded fix
Amnesiadefaults to workspace team_agent_lab, not the project's — agent starts with no historypin the project's workspace everywhere
Metered Bedrockbare model: claude-code fails validation → falls back to AWS Bedrock (billed!) when creds are presentstructured {provider: anthropic, default: <concrete>}
Alias 404default: claude-code is an alias, not a real API model id → 404a concrete model id (e.g. claude-opus-4-8)
Scratchpad fragilityinstall in a temp venv that gets cleaned → launcher dies laterone durable install on PATH
Clobberingthe old preset overwrote .mcp.json and force-installed the team ontologydetect + merge; never overwrite
Methodology-blindnessthe agent has no idea about CG, precedent-first, or governed actionsship an auto-loaded working agreement

The subscription-vs-metered ones (2, 3) aren't just bugs — they're an A2 violation (our hard rule: subscription auth only). See §7.

corethe shape

The model: one CLI + a tiny .cohermes.yaml

Stop generating bespoke scripts. Instead: one durable cohermes command on PATH (the engine that holds all the trap-fixes), plus a small committed .cohermes.yaml per project that makes the repo self-describing. The command reads the config and does the right thing from any onboarded directory.

The engine carries the knowledge; the project declares a few facts; the backends are shared.
# .cohermes.yaml — committed to the repo
workspace: investigator
cg_url:    http://localhost:9621
model:     {provider: anthropic, default: claude-opus-4-8}
ontology:  generate          # generate | existing | team
flavors:   [hermes, claude]   # which run modes to enable
developer: dima
CommandDoes
cohermes onboard [dir]the universal onboarder — §4
cohermes hermes [task]run the Hermes runtime flavor — §5
cohermes agent [task]run the Claude Code + orient-brief flavor — §5
cohermes doctorverify wiring, auth (A2), CG reachability, model resolution

This retires the per-project cg/hermes + cg/cohermes-agent scripts: their logic moves into the engine; the repo keeps only .cohermes.yaml + the working agreement.

flowthe onboarder

cohermes onboard — non-destructive by default

Idempotent; every write is additive or a merge. Re-running is safe.

1Preflight. CG reachable? Claude logged in on a subscription (A2)? A durable cohermes install present? Each is checked and, where possible, offered a fix — never assumed.

2Detect. Does the workspace already exist and hold records / an ontology? Does the repo already have a .mcp.json? Existing state is preserved and reported, not overwritten. This is the investigator lesson made structural.

3CG-side setup. For a fresh workspace, run CG's conversational onboarding wizard (/onboard, /onboard/chat, /onboard/apply) to generate a project-fit ontology + action set. If the workspace already has one, keep it (§6). Optional curated git/docs backfill.

4Client wiring — all additive:

  • write .cohermes.yaml;
  • generate the project-local Hermes home (.hermes/) with the structured model + the context-graph MCP pinned to the workspace via the LIGHTRAG-WORKSPACE header;
  • merge context-graph into any existing .mcp.json (Claude Code flavor) — never overwrite;
  • write the working agreement — .hermes.md (Hermes) and/or CLAUDE.md (Claude Code) — pointing at the CG protocol;
  • gitignore the local home.

5Verify. A read-only smoke test — orient / precedent query — confirms the agent reaches the right workspace on the subscription, then prints a report and next steps.

Non-destructive invariant
Onboarding never overwrites an existing .mcp.json, never wipes an ontology, and never writes a metered key. If it can't do something safely, it reports and asks — it doesn't guess.
userunning it

cohermes run — the two flavors, one config

Both flavors read .cohermes.yaml, pin the workspace, and scrub metered-billing env. Crucially, in both the actual reasoning runs on the real claude binary (first-party, plan limits) — they differ only in what surrounds it.

FlavorCommandWhat runsUse when
Claude Codecohermes agentthe real claude binary directly (claude -p / interactive), pre-briefed from the brain — first-partyquick, brief-driven tasks
Hermes runtimecohermes hermesa developer-local coordinator model orchestrates real claude subprocesses (the claude-code skill, claude -p/tmux) — first-party work, local coordination — plus Hermes's tools / memory / gatewayautonomous / persistent / multi-agent

The heavy tokens are always the first-party claude subprocess (plan limits). The Hermes coordinator's own turns run on a developer-chosen local model, so they cost nothing and never meter.

D2the memory shape

Ontology: conversational generate, with detect-and-preserve

Default = conversational generation. For a new project, onboarding runs CG's LLM wizard: a short interview about the project's objects and moves, and it generates a project-fit ontology + governed action set — richer and more accurate than forcing one shape on every project.

But generation is gated by detection. If the workspace already has an ontology and recorded history (investigator's ChangeRequest / ADR / Task / Module + investigator_dev actions), the onboarder preserves it — the wizard confirms/augments rather than forking a parallel decision model. The ontology: field records intent:

ontology:Behavior
generaterun the CG wizard (default for a fresh workspace)
existingreuse what's already in the workspace, untouched (investigator)
teaminstall the standard 4-artifact ontology (Decision→Task→Commit→Review)
A2the hard rule

Subscription-only — and the two ways Hermes touches Claude

A2 — agents run on the Claude subscription, never metered — is guaranteed by keeping all real reasoning on the first-party claude binary. A hard-won distinction (2026-07): Hermes can reach Claude two ways, and only one is first-party.

MechanismBilling
✅ the claude binaryHermes runs claude -p/tmux as a subprocess (the claude-code skill); or cohermes agent spawns it directlyplan limits — genuine first-party
❌ coordinator-as-ClaudeHermes calls the Messages API with a claude-code (external, cli) user-agent using your OAuth tokenextra usage — metered
The 2026-07 policy change
Anthropic now routes third-party apps using your subscription OAuth token through "extra usage" (a paid overage), not your plan limits. So we never route real work through path ❌ — that's the whole point of keeping the worker on the actual binary.

The design splits the roles accordingly:

  • Worker = the real claude binary — all actual coding/reasoning. First-party, plan limits.
  • Coordinator = a developer-local model — Hermes's own orchestration turns run on a local model (llama.cpp / Ollama), so they never meter. This is a per-developer setting (each dev runs their own hardware), kept in a gitignored override — not the committed .cohermes.yaml — and easy to change: cohermes coordinator set --base-url … --model …. Onboarding auto-detects a local endpoint. (Note: the coordinator model needs a ≥64K served context window — Hermes's floor.)
  • launchers scrub ANTHROPIC_API_KEY + cloud vars; onboarding refuses to write a metered key, and cohermes doctor fails loudly on one.
D4optional, later

The UI configurator

The Hermes dashboard already ships Config / Models / Mcp pages. A future cohermes onboard --ui can drive the same onboarding engine through a small wizard for people who'd rather click than type — workspace pick, ontology interview, flavor toggles, a live A2/CG health panel.

But it's a convenience layer, not the core. We build the CLI onboarder first and keep the UI as a thin front end over it — so there's exactly one implementation of the six trap-fixes.

Build plan

P1Engine + config. Ship the durable cohermes command, the .cohermes.yaml schema, and cohermes onboard / hermes / agent / doctor — with the six traps and the non-destructive detect+merge encoded once.

P2Conversational ontology. Wire onboard into CG's /onboard* wizard for fresh workspaces; detect-and-preserve for existing ones.

P3Migrate investigator to .cohermes.yaml (retire cg/hermes + cg/cohermes-agent) as the first real regression test — same behavior, no bespoke bash.

P4Optional UI — a dashboard wizard over the onboard engine.

First proof
Success = cohermes onboard on a brand-new empty project produces a working, subscription-only, workspace-pinned, methodology-aware setup with zero hand-edits — and re-running it on investigator changes nothing.
