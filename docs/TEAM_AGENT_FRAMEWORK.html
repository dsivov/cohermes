The plan

Scope & method

The idea in one sentence: take a Hermes-style autonomous Claude Code agent, give one to every developer, and wire them all to a shared Context Graph so the team's knowledge — code, decisions, reviews, tasks, architecture — lives in one place every agent can read and write.

This is not a green field. Context Graph already ships most of the substrate, and several of your docs already circle this idea — AGENTIC_DEVELOPMENT.html ("AI agent development, remembered"), AGENTIC_PROJECT_GRAPH.html ("CG as the methodology actor", which already sketches a sync bus and roles-as-identities), and the platform work (PLATFORM_ARCHITECTURE.html, GOVERNED_WORKFLOW_PLATFORM.html). So this doc's job is to name what is genuinely new — the Hermes local-agent loop and the team fabric — and to build on those docs, not re-propose them.

Repo topology — where the code lives

A deliberate split, decided up front (A1): the framework is not built inside the Context Graph repo. Instead:

Framework fork of hermes-agent
A fork of the MIT-licensed hermes-agent (Python agent runtime, 40+ tools, terminal backends, learning loop). This is our codebase — we inherit the orchestration and swap the substrate.
Backend Context Graph, as-is
CG runs as the shared service (MCP + REST). We add modules to it only when we hit a genuine gap (identity, review ingest, sync) — CG stays clean, reusable, not coupled to the agent framework.
This doc CG docs family
The design log lives in CG/docs as a companion to the agentic/platform docs it builds on. The implementation lives in the fork.
The sharpened thesis — hermes already remembers; we make it remember together
Forking revealed the cleanest possible framing: hermes-agent already ships a memory system — but it's per-user, cross-session recall for one operator. Our entire contribution is to replace that private memory with the team's shared Context Graph. Same runtime, same tools; the difference is that every agent's "memory" is now the whole team's living project graph. That single swap is the framework.
The core insight

The gap Hermes-agent leaves — and why CG fills it

NousResearch's hermes-agent is an excellent orchestration layer: it drives a single Claude Code CLI (print mode for one-shot tasks, tmux/PTY for interactive), manages permissions and tool whitelists, handles trust prompts, and can spin up parallel worktree instances. What it does not solve is where the agent's knowledge lives.

Concept · the amnesia problem, multiplied by the team
A Claude Code agent has excellent in-session recall and near-total between-session amnesia — its context is a local CLAUDE.md and a resumable transcript. That's already the problem AGENTIC_DEVELOPMENT.html names for one developer. Now scale it to a team: five developers, five agents, five private CLAUDE.md files. Developer B's agent has no idea why Developer A's agent chose an approach yesterday, what was rejected in review, or which task is in flight. Every agent re-derives context that another already established, and they drift apart. Hermes orchestrates each agent beautifully; it gives them nothing to share.

Hermes answers "how do I drive an autonomous coding agent?" Context Graph answers "what do a team of them remember together?" The framework is the marriage of the two.

CG is unusually well-suited to be that shared brain because it isn't a passive index — it's built to be a methodology actor (AGENTIC_PROJECT_GRAPH's framing): it stores decisions with their reasoning (the quadruple (h,r,t,rc)), holds current project state, keeps agents in sync, and can refuse process-breaking moves via its governance layers. That's exactly the substrate a fleet of agents needs.

The shape

The fleet model

Each developer works in their own environment — own machine, own repo checkout, own Claude Code. A Hermes-orchestrated agent runs alongside them. Every agent connects to the same CG workspace (one project = one shared graph), so reads and writes flow through a common knowledge fabric.

Ground truth · the built shape

Architecture (as built)

The topology after M1–M4: what runs on each developer's machine vs what is the one shared backend — and, specifically, why cohermes ships its own MCP server.

Why cohermes ships its own (local) MCP server — the ① you asked about

CG already has an MCP server (context-graph, 12 tools). So why a second one? Because the operations agents actually need are richer than CG's primitives, and we chose not to modify CG:

QuestionAnswer
Why not just CG's MCP?CG's record_decision writes a trace only. cohermes needs composite ops — the node+trace dual-write (D10), the chain writers + links, ingest_context, record_learning — each is several CG calls orchestrated. Those are cohermes logic, not CG's.
Why not add them to CG's server?A1: keep Context Graph unmodified and reusable. Baking cohermes tools into CG would couple them and need a CG redeploy. The extension lives in cohermes.
Why local (stdio), not a hosted cohermes server?It holds no state — it just orchestrates calls to the shared CG over HTTP. So there's nothing to host or secure: the agent's .mcp.json launches it as a subprocess. Zero shared infra; the only shared thing is CG.
What does local buy us?It's configured with this developer's identity + workspace (env), so it naturally stamps attribution (D2) — no per-request principal plumbing. And it matches Hermes's per-developer, run-locally model.
The one-line mental model
Context Graph is the shared brain (all shared state). The cohermes MCP server is a thin, per-developer adapter — it gives each agent richer, identity-stamped tools that compile down to CG HTTP calls. Sharing happens at CG; the local server is just how an agent reaches it well.

Ground truth · the gaps

The gaps to build — the "more than one agent" problems

Everything the solo flow needs exists. The genuinely new work is the team dimension: the four things that only become problems when many agents share one graph.

1 · Per-developer identity gap
Today RBAC is role-based, and MCP carries no authenticated identity at all — invoke_action passes role=None (mcp_server.py:523-528). For a team you need who: which developer's agent made this decision, touched this task, approved this review. Attribution and per-user RBAC are the missing layer.
2 · Code-review / PR ingestion gap
Backfill ingests commits + docs + source, but not PR/review threads. Reviews are where the richest "why" and "why not" live — a net-new connector (GitHub/GitLab → decision edges + review objects).
3 · Task assignment & board partial
Lifecycle FSMs + actions can model task state, but there's no notion of assigning a task to an individual or a board view across the fleet. Builds on gap 1 (identity).
4 · Cross-agent sync deferred (D5)
When agent A records a decision, should agent B learn of it proactively, or just discover it next time it queries? Hermes source audit settled this: hermes has no native agent↔agent bus / shared memory / event subscription — its one reusable sync primitive is the cron scheduler (--script + --deliver). So v1 is pull-only (a scheduled watcher diffs the CG API); true push would be a CG-side build, deferred to the team phase.
Not to be misread — writes are always on
Pull-only (D5) is about awareness, not writes. Two separate flows: (1) every agent writes into CG continuously during development — recording decisions, advancing tasks, ingesting commits and reviews (the whole point; the graph gets richer with every dev action). (2) How a teammate's agent learns of those writes is the pull/push question — and there we chose pull (query fresh) over push (get pinged). The graph is always being updated; pull-only only means peers discover each other's updates by querying rather than by notification.
Scoping honesty
Gaps 1–3 are concrete and bounded. Gap 4 (sync) is the deep, open one — and it's where the interesting design lives (pull vs push, conflict when two agents touch the same object, staleness). It's tempting to start there; it's probably wiser to start where identity + a real shared artifact prove the fabric works, then earn our way to sync.
The new artifact

The local agent loop — the Hermes side

The part that is genuinely ours to design: the loop each developer's agent runs. Hermes provides the autonomy harness; CG provides the shared brain; the loop is the contract between them. In spirit it's the "three habits" the playbook already preaches (workspace_routes.py:249-382), turned into an explicit agent skill.

Delivery question embedded here

Hard constraint · A2

Subscription auth, never metered API billing

A team-wide cost requirement: every agent must run Claude Code on a Pro/Max/Team subscription (flat per-seat), never pay-per-token API billing — in both tmux and claude -p modes. Verified against both the hermes skill and the Claude Code auth docs: this is supported, preserved across modes, and — crucially — enforceable.

Good news — hermes doesn't fight us
The hermes skill never injects an API key: its launch commands (claude -p '…', tmux send-keys … 'claude' Enter) pass no credentials and rely entirely on the machine's prior login state. Print mode and interactive mode resolve the same stored credential, so a subscription login works identically in both. The one exception to avoid: hermes --bare mode requires ANTHROPIC_API_KEY and skips OAuth.
Concept · the auth precedence chain (and the footgun)
Claude Code picks its credential by a strict precedence, highest first: Bedrock/Vertex/Foundry env vars → ANTHROPIC_AUTH_TOKEN → ANTHROPIC_API_KEY (metered!) → apiKeyHelper → CLAUDE_CODE_OAUTH_TOKEN (subscription) → OAuth /login (subscription). The footgun is stark: a stray ANTHROPIC_API_KEY in the environment silently wins over a logged-in subscription, and in -p mode it's used with no prompt. A shared team .env or shell profile that exports it is the single most dangerous pattern.

Team billing notes
Each developer authenticates with their own subscription seat (no token sharing — one seat can't be used by two people at once). Per-seat rate limits (TPM) scale down as the team grows and reset on rolling 5-hour + weekly windows, shared with their Claude chat usage. A fleet of always-on agents could exhaust a seat's window — so agent pacing / concurrency caps become a real design concern, not just billing hygiene. This feeds Q1 (solo-excellent first lets us learn one seat's real throughput before scaling the fleet).
The philosophy

Why this is your "constrained expert system", for developers

Your AGENTIC_EXPERT_SYSTEM.md and GOVERNED_WORKFLOW_PLATFORM.html argue the same thesis from a different angle: an AI agent should be a constrained system — bounded by an ontology, rules, and typed building blocks — not a free-wheeling "natural-language compiler". A team of coding agents is exactly that thesis applied to software development itself.

The connection
A lone coding agent with bypassPermissions is the "general-purpose compiler" your notes warn against — it invents structure, drifts from methodology, and forgets the constraints. Wire that same agent to a shared CG and it becomes constrained by the team's ontology and rules: it must use governed actions, its decisions pass the rules gate, its moves respect lifecycle state, and its context is the team's real graph, not its own guesses. The graph is the DSL; the agents may only wire existing typed blocks. That's the governed-workflow idea, with the software project as the domain.

The framework isn't just shared memory — it's shared constraint. The graph is what keeps five autonomous agents building one coherent system instead of five divergent ones.

The record · round 1

Round-1 questions settled

All five are decided (D1–D5 in the log below), each after checking what hermes provides natively first (the reuse-first principle). Kept here for the record.

Q1 MVP boundary — solo-excellent first, or team-first?
Polish the single-developer Hermes+CG loop until it's genuinely great (fast path to something usable), or build the team dimension (identity + a shared artifact) from the start (the actual point, but heavier)? My rec: solo-excellent first — it's the foundation every fleet feature stands on, and it's demoable in days.
Q2 Identity — per-developer now, or roles for v1?
Do we add real per-developer identity (who did what, per-user RBAC, MCP-carried principal) now, or lean on CG's existing role-based model for v1 and defer individual identity? My rec: roles for v1, but thread a lightweight developer attribute through decisions/tasks immediately so attribution exists even before full per-user auth.
Q3 The loop's form — Hermes skill or CG-native preset?
Ship the loop as a hermes-agent SKILL.md (drops into their ecosystem, print/tmux modes) or as a CG-native agent preset (bootstrap-generated CLAUDE.md + slash-commands, Hermes-agnostic)? My rec: CG-native preset first (works with any Claude Code, reuses bootstrap), and a thin Hermes SKILL.md wrapper as a follow-on so it also runs headless/CI.
Q4 First shared artifact — decisions, reviews, or tasks?
Which team-knowledge type do we make excellent first? Decisions (already strongest in CG), code reviews (richest "why", but needs the new PR connector), or tasks (needs identity + board)? My rec: decisions — it's built, it's the highest-value shared knowledge, and it proves the "one agent records, another benefits" loop end-to-end with least new plumbing.
Q5 Sync model — pull-only for v1, or push?
Do agents just query fresh each task (pull — simple, stateless, good enough early) or do we build proactive push notification when the graph changes (the "sync bus" — powerful, but the hard open problem)? My rec: pull-only for v1; design the record format so a push layer can be added later without reworking it.
Round 2 · the spec

Build plan

Translating the decisions into a sequenced plan. Everything is solo (D1), reuse-first (extend hermes + CG before building), and aimed at one target: a demo that proves the concept.

Definition of done — what "concept proven" means

The demo we're building toward
A single developer's forked-hermes agent, over a work session, produces a CG graph where you can trace the whole chain — a Decision motivates a Task, the Task is implemented by Commits, the Commits are reviewed in a code Review, the Review enacts a Decision — and then a later agent session queries that chain and reuses it instead of re-deriving it. Non-negotiables it must show: it ran on subscription auth (A2), it wrote to CG continuously during the work (not just at the end), and it pulled fresh context at task start (D5). That single traceable-and-reused chain is the concept.

Per D4, the ontology comes first: it's what makes four artifact types into one connected brain rather than four silos. Reuse-first: extend the existing agentic-dev preset (ontology/rules/lifecycle already shipped, per AGENTIC_DEV_WALKTHROUGH.html) rather than author fresh.

Dev A (dima)'s agent recorded a decision: "Use OAuth 2.0 for the webhook connector". Dev B's pull-watcher (baseline taken before A acted) then reported the change — awareness without a push bus. Dev B (sarah)'s separate agent, tasked with token refresh, first search_precedents → found dima's decision, quoted it, and named dima as its author — then recorded a follow-on (approved_by sarah) whose rationale explicitly "builds on the team decision (approved_by dima) to use OAuth 2.0". That is the whole point: a team of agents building one system, not many.
Honest edges
The agents write via CG's MCP record_decision (edge-trace only), so their entries are traces, not full Decision nodes (the watcher lists them as generic nodes). A cohermes MCP tool doing the node+trace dual-write (D10) is the clean next refinement. And identity is attribution, not authenticated per-user auth — that remains deferred.

The two-layer brain — the cohermes MCP server ✓

A gap surfaced: the graph had no chunks — we'd only written structured nodes/edges, never ingested prose. A decision-log is not a project memory. Fix: a self-contained cohermes MCP server (cohermes/cg/mcp_server.py, a local stdio server the preset adds beside context-graph) giving agents both layers:

Spine — record_decision (node+trace, D10), record_task/commit/review, link
agents now build typed Decision nodes and the full chain themselves, not just traces
Flesh — ingest_context
prose (architecture, design, status) → CG chunks + extracted entities → agents can explain the system
Proven via a real agent
An agent called cohermes record_decision → a real Decision node (by dima); and ingest_context → the architecture note processed to 1 document + chunks (index was empty before). A follow-up query "explain the architecture of cohermes" now returns a rich, synthesized answer (both layers, node types, edge semantics) where before there was nothing to retrieve. The brain has depth now, not just a decision list.

What flows into the brain — curated, not everything (D11–D13)

The brain is a curated record, not a data lake. High-value, durable, non-derivable knowledge is promoted through a curation gate — each entry stamped with author, confidence, and validity. Raw code, diffs, and unfiltered agent memory stay out: they're already in git, or they're noise that would dilute retrieval.

record_learning + make-the-brain-real + orient ✓

Three moves that turn the brain from a test fixture into something that reduces drift.

record_learning (D12/D13)
agents promote curated learnings as attributed, confidence-scored Insight nodes + retrievable prose, with active/superseded validity. Proven: an agent promoted a learning; a query retrieves it verbatim.
Curated backfill (D11)
dogfood — cg/backfill.py ingested cohermes's own 13 module purposes (docstrings) + design docs as prose + Module nodes. Not a repo dump; the durable why.
orient (read-from-brain)
cg/orient.py + tool brief an agent at task start: relevant decisions + active learnings + architecture context — so it begins aligned.
The drift-reduction loop, closed — proven with a real agent
Before building a new connector, an agent called orient and the brief surfaced sarah's "use fixture-first connectors" decision and the "MCP needs PYTHONPATH=repo root" learning. The agent reused them, cited them, and stayed within the existing pattern — it began with the team's current understanding instead of re-deriving. That's the whole point of a shared brain: agents start aligned and end aligned. (Diagram updated: writes → brain, orient ← brain, backfill seeds it.)

Deeper Hermes integration — a native runtime hook ✓

So far agents reached the brain via the cohermes MCP tools (Claude-Code-level). This wires cohermes into Hermes's own agent lifecycle via its designed extension point — an event hook (cohermes/cg/hermes/: HOOK.yaml + handler.py, deployed to ~/.hermes/hooks/cohermes/). On agent:start it (1) runs the A2 subscription-auth preflight and (2) orients the agent from the shared brain (read-from-brain) — the Hermes-native counterpart to the MCP tools.

Smoke-tested through Hermes's real hook machinery
We stood up a minimal Hermes locally and fired the hook through its actual HookRegistry — not a mock: it discovered and loaded the hook ([hooks] Loaded hook 'cohermes' for events: ['agent:start']), and registry.emit("agent:start", ctx) ran our handler → auth "OK subscription (max)" + a real 1769-char orient brief written from the shared brain. So Hermes discovers, loads, and fires cohermes exactly as it would in production. One honest limit remains: Hermes hooks are side-effect-only (returns discarded), so the brief is delivered (to $HERMES_HOME/cohermes/brief-<session>.md) rather than injected into the prompt — automatic prompt-injection would need a change to Hermes's context assembly (invasive, deferred).
Next milestone — the real live test
Build + install cohermes properly and run it live: Claude Code + Hermes, two developers, CLI and UI — the end-to-end proof that validates the hook (and the whole stack) in a real deployment.

Packaging

What the live test still needs (real machines)
A second developer's machine, and — for the Hermes-orchestrated flow — a configured Hermes (its own env + a model provider). Both are real-deployment steps; the runbook lays them out. The Claude-Code-direct flow (preset + cohermes MCP) is already proven and needs only the two devs.

cohermes-agent — automatic read-from-brain at launch ✓

Taking the Hermes smoke test one step further exposed a clean resolution to the "hooks can't inject prompt context" limit: inject the brief at the launch layer via Claude Code's own --append-system-prompt — no Hermes-internals change needed. Productized as cohermes-agent (cohermes/cg/run.py): auth preflight → orient(task) → launch Claude Code with the brief injected.

Proven — a no-tools agent, aligned automatically
Given a project with no MCP tools and told not to look anything up, an agent launched via cohermes-agent answered "how should we test a connector?" by reusing the team's recorded decisions verbatim — fixture-first (sarah), OAuth 2.0 + 5-min refresh (dima/sarah), retry cap, local stdio MCP. It started aligned purely from the injected brief — the read-from-brain payoff, now automatic and Hermes-agnostic.
Integration mode + setup guide
Decided: -p (print) as the primary mode — one call per task, brief injected at launch, cross-task continuity via the brain (not Hermes-session-history, which cohermes deliberately doesn't depend on); tmux optional for live multi-turn sessions. Full setup guide with the mode comparison + context-flow diagram: cohermes/docs/SETUP.html.
